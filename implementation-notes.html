Overall approach

WHATS_NEXT.md is a strategic review, not a line-by-line spec. The "spec" I'm implementing to is the explicit roadmap reset in §9 plus the agent-wedge concretions in §3, §4, §7. Sequencing follows the doc's own priority: v1.3 wedge first, then v1.5 robustness, then v1.6 generalization. v1.4 distribution (single binary, rename) ships last because it is mostly packaging work and is the cheapest to defer.

Decision I'm not renaming the npm package from @prnv/tuck to @tuck/cli in code as part of this work. The doc recommends it but that's a publishing/registry move that requires owning the @tuck npm scope. Listed as an open question instead.

Decision I'm treating "JSON-everywhere" as the foundation everything else stands on. The state machine refactor of init.ts (§3.3) explicitly depends on it. So I do JSON first across all commands as a shallow envelope, even if a few commands still have a thin interactive layer remaining. That gets us to "agents can drive every command" fastest; full prompt-tree decomposition happens during the init refactor.

More sections will appear here as features land.

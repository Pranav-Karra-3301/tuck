Overall approach

WHATS_NEXT.md is a strategic review, not a line-by-line spec. The "spec" I'm implementing to is the explicit roadmap reset in §9 plus the agent-wedge concretions in §3, §4, §7. Sequencing follows the doc's own priority: v1.3 wedge first, then v1.5 robustness, then v1.6 generalization. v1.4 distribution (single binary, rename) ships last because it is mostly packaging work and is the cheapest to defer.

Decision I'm not renaming the npm package from @prnv/tuck to @tuck/cli in code as part of this work. The doc recommends it but that's a publishing/registry move that requires owning the @tuck npm scope. Listed as an open question instead.

Decision I'm treating "JSON-everywhere" as the foundation everything else stands on. The state machine refactor of init.ts (§3.3) explicitly depends on it. So I do JSON first across all commands as a shallow envelope, even if a few commands still have a thin interactive layer remaining. That gets us to "agents can drive every command" fastest; full prompt-tree decomposition happens during the init refactor.

v1.3 #1 — JSON-everywhere

What landed

  • src/lib/jsonOutput.ts: shared envelope { ok, command, data | error, warnings? } plus isJsonMode() gate.
  • TuckError.toJSON() with stable field order and per-class exitCode.
  • --json flag added to: scan, status, list, doctor (preexisting), and new: diff, restore, undo, encryption status/encrypt-file/decrypt-file, context *, mcp tools, preset *, bundle *.
  • Early-detection of --json in src/index.ts so the UI layer + error handler can silence themselves before any command-specific action runs.

Decisions

Decision Existing --json outputs on status/list/scan/doctor already shipped with their own shapes. To avoid breaking 800+ tests they were left as-is — new --json outputs go through the envelope. A future minor release should migrate the legacy outputs (one schema everywhere is the goal).

Decision sync and apply and init commands gained --json support indirectly: bundle agent added --bundle on apply with envelope-aware output; merge UI throws a structured MergeConflictsError with exit code 3 in non-interactive mode.

Open question The sync happy-path JSON envelope isn't yet wired — it still uses prompts intros even in --json. The plumbing is in place (isJsonMode()); ship the gating in the init refactor task.

v1.3 #4 — --plan mode

Implemented on preset apply and restore. The semantic is: emit the operation as JSON (or text, with --dry-run) and exit before mutating anything.

Tradeoff --plan and --dry-run are intentionally distinct: --plan is the agent-facing JSON; --dry-run is the human-facing text. They share the same code path internally so they don't diverge.

Open question sync doesn't yet have --plan because the change set is more involved (need to compute new tracked files separately from modifications). Listed in the init refactor work.

v1.3 #2 — tuck context

New top-level surface for AI agent configs across home and per-repo scopes. Subcommands: add, list, scan, sync, apply.

Decision Repo-scope detection walks up from the file looking for a .git directory; if found, the file is repo-scoped (even if it lives under $HOME). This matches the WHATS_NEXT.md ask in §6.2.

Decision A separate context.json sibling to .tuckmanifest.json holds context entries. Mixing them in one manifest would have required a schema migration on every existing install; this is cleaner and the on-disk layout under context/home/ and context/repos/<repo-key>/ makes the two scopes self-evident.

Deviation tuck context apply currently materializes only home-scoped files. Repo-scoped apply is deferred to land alongside templating (§5.1) so we don't render a CLAUDE.md from another repo without resolving its template variables to this machine first.

Open question The slugifier used for destination keys is intentionally lossy. If two paths slugify to the same key (rare but possible), the second add overwrites. Worth revisiting once the registry has real users.

v1.3 #3 — tuck mcp serve

Hand-rolled stdio JSON-RPC MCP server. Exposes 6 tools: list_tracked_files, status, scan, context_list, context_add, classify_agent_file.

Tradeoff Chose hand-rolled JSON-RPC over @modelcontextprotocol/sdk to keep tuck's install footprint small. The SDK adds material weight to a CLI that may be installed on minimal boxes. The handler is ~70 lines of glue and could be swapped for the SDK any time without API churn.

Open question Resources (vs tools) aren't exposed yet. Returning the tuck manifest as an MCP resource might be a nicer agent UX than a tool call. Decide once an MCP client actually consumes this.

v1.4 #1 — Patterns-as-data registry

DOTFILE_PATTERNS is no longer the source of truth at runtime. src/lib/patternsRegistry.ts loads JSON from templates/patterns/*.json first, then merges ~/.tuck/patterns/*.json overrides on top. The in-code constant remains as a build-time fallback so dev builds without the data dir still work.

Decision Format is one JSON file per category. Easier to PR a new tool than threading additions through a single mega-file.

v1.5 #1 — Templating

Small handlebars-flavored DSL + comment-marker style. {{var | default "x"}}, {{#if os == "darwin"}}…{{/if}}, and # tuck:if os=darwin / # tuck:endif.

Decision No arbitrary expressions, no function calls, no template inheritance. The grammar is deliberately small so the sandbox boundary is trivial to audit. Variables come from a frozen context: os, arch, user, home, hostname, ci, plus env.X.

Decision Rendering happens at restore time, never at storage. The doc was explicit about this being a chezmoi misstep; we match.

v1.5 #2 — File-level encryption

AES-256-GCM + PBKDF2-HMAC-SHA256 (200k iterations). Magic header TCKE1 + salt(16) + iv(12) + tag(16) + ciphertext. 14 tests cover round-trip, tamper, empty, large (≥1 MiB), wrong-password.

Deviation The spec mentions age by name. We use AES-GCM via Node's built-in crypto module instead. Reasons: no new dependency, no native binary requirement, no key-management surface to maintain. The on-disk format is documented and stable; users who want age compatibility can layer it externally.

Decision User-facing surface is currently tuck encryption encrypt-file / decrypt-file. Wiring tuck add --encrypt end-to-end (decrypt at restore, transparent for the user) is a follow-up because it touches the manifest's encrypted field and the sync/restore pipeline.

v1.5 #3 — Merge UI

git pull --rebase failures now route through src/lib/mergeConflicts.ts: detect conflicts, drive an interactive resolution (keep ours / theirs / edit / abort) via src/ui/merge.ts, snapshot before+after, then git rebase --continue.

In --json mode or a non-TTY, the merge UI is bypassed and a structured MergeConflictsError (exit code 3) is thrown so agents can detect and surface the situation cleanly.

v1.5 #4 — --dry-run

restore and undo already had --dry-run; preset apply gained it; --plan covers the agent side everywhere it's wired. sync and add still lack a real --dry-run — listed in the init refactor task.

v1.6 #3 — Presets

Format defined; 5 starters shipped (minimal, modern-cli, claude-code, agent-everything, starship) under templates/presets/. CLI surface: tuck preset list|show|apply|publish.

Decision Presets are JSON, not YAML. YAML would require a runtime parser dependency or a hand-rolled parser; JSON has none of that complexity. The doc shows YAML in §7.1 as an example; spec said "JSON or YAML" — we picked JSON for v1 and listed YAML as an open question.

Tradeoff Presets that would overwrite a user's ~/.zshrc (modern-cli, starship) instead write to a sidecar (~/.config/tuck/<name>.zsh) and the postApply hook nudges the user to source it. Safer default; users who want full replacement can use minimal.

Open question Should there be a remote preset registry (e.g. tuck preset apply tuck-sh/claude-code)? The spec leaves this open; the local registry is enough to ship v1.

v1.6 #1 — Bundles

Each tracked file now has a bundle field (default "default"). Migration is transparent in loadManifest: legacy manifests without bundles get a synthetic default entry plus all files reassigned to it. tuck bundle list|create|rm|assign is the surface; tuck add --bundle and tuck apply --bundle route files in and out.

v1.6 #2 — Repo-scoped tracking

Wired via tuck context add. Files inside any git working tree are tracked relative to that repo's root, not $HOME. The manifest stores the original repoRoot so apply on a different machine can rehome the file at the target's matching repo.

Open question When the same repo is cloned at different paths on different machines, how does context apply map repos? Today: lookup-by-basename or explicit --repo flag (not yet implemented). Document a stable identifier (origin URL?) before this gets public usage.

v1.4 #2 — Single binary build

Not yet wired. The Bun-compile path is a follow-up; tsup still produces the ESM CJS build. See §3.4 — it's a 1-day change but I prioritized the agent wedge.

v1.5 #5 — init.ts state machine

Not yet refactored. The 1747-line file remains a procedural script; agents bypass it via tuck context add and the MCP server. A full state-machine refactor is the next piece of structural work, and it unlocks sync --plan, sync --json, and a non-interactive tuck init.

Open questions (consolidated)

  • npm package rename to @tuck/cli — not done; requires owning the @tuck scope. decide before announcing
  • Bun single-binary build vs. staying Node-only — not implemented yet; want your call on signing infrastructure for macOS.
  • YAML preset format support — punted; JSON only for v1.
  • Remote preset registry — punted; local-only for v1.
  • Migrate legacy --json outputs (status/list/scan/doctor) to the envelope — needs a coordinated bump and is breaking for agents that already consume them.
  • The aspirational manifest fields (encrypted, template, permissions) are now real for encrypted/template but permissions is still unused. Either wire it in or remove it.
  • Identifier for cloning the same repo at different paths across machines — see repo-scoped.
